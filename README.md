# Spaceflight–Disease Network Separation Pipeline

A computational pipeline that uses **network biology** to quantify how spaceflight-responsive genes relate to known disease modules within the human protein–protein interactome (PPI). The core metric is a permutation-based separation z-score (`separation_z_score`) from [NetMedPy](https://github.com/menicgiulia/NetMedPy), applied between a combined spaceflight gene module (JAXA CFE + NASA Twins + Inspiration4) and hundreds of curated disease gene sets.

---

## Pipeline Overview

```
Notebook 1: Disease + Spaceflight Module Curation
  ├── Download MONDO ontology
  ├── Load & map OMIM (genemap2) gene-disease associations
  ├── Load & map GWAS Catalog associations
  ├── Build PPI interactome from PCNet 2.2
  ├── Filter: ≥20 genes per disease module in PPI
  └── Build spaceflight module (JAXA + Twins + Inspiration4) → inject

Notebook 2: Separation Pipeline
  ├── Benchmark: auto-configure worker count & fit runtime model
  ├── Compute all-pair shortest-path distances (cached)
  ├── Parallel Ray screening → per-disease checkpoints
  ├── Merge checkpoints → separation_results
  ├── Multiple testing correction (BH-FDR + Bonferroni)
  └── Empirical FDR calibration

Notebook 3: Visualization
  ├── Z-score landscape (ranked scatter)
  ├── Size-stratified distributions
  ├── Top-N separation bar chart
  ├── Geometric vs. statistical separation scatter
  ├── P-value distributions
  └── Gene set size bias check
```

## Running the Pipeline

Run the three notebooks **in order**:

### Step 1 — Disease Module Curation
```bash
jupyter lab 1_disease_module_curation.ipynb
```
Run all cells top to bottom. This produces:
- `data/processed/ppi/ppi_network.pkl` — cleaned interactome
- `data/processed/gene_modules/gene_modules.pkl` — disease + spaceflight gene sets
- Audit CSVs in `data/output/`

### Step 2 — Separation Pipeline
```bash
jupyter lab 2_separation_pipeline.ipynb
```
Run all cells. The benchmark cell auto-configures worker count and runtime model. The screening cell is restartable — it skips already-completed checkpoints. This produces:
- `data/output/sp_distance.pkl` — cached all-pair shortest-path distances
- `data/output/benchmark.json` — worker config and runtime model
- `data/output/checkpoints/<disease>.pkl` — per-disease results
- `data/output/disease_separation_results.pkl` — merged results
- `data/output/csv/separation_summary.csv` / `separation_significant.csv`

**Optional: run screening in the background** (useful for large jobs):
```bash
python run_screening.py
# Monitor progress:
tail -f data/output/screening.log
```
`run_screening.py` is auto-generated by Notebook 2 and mirrors the interactive screening cell exactly.

### Step 3 — Visualization
```bash
jupyter lab 3_spaceflight_disease_visualization.ipynb
```
Reads `data/output/csv/separation_summary.csv`. Produces six publication-ready plots in `data/output/plots/`.

## Key Design Decisions

**Gene ID mapping** uses a local NCBI `Homo_sapiens.gene_info` file rather than live API calls. This makes the pipeline fully reproducible offline and avoids rate-limit failures during large batch runs.

**Checkpointing** saves each disease result to an individual `.pkl` file immediately after completion. If the pipeline is interrupted, re-running picks up exactly where it left off.

**Runtime model** is fit at benchmark time (linear or quadratic, selected by R² improvement). The model drives LPT-scheduled ETA estimates shown in the screening log.

**Multiple testing** applies both BH-FDR and Bonferroni correction. Classification into Overlapping / Neutral / Separated uses FDR < 0.05 combined with the sign of the z-score.

## Troubleshooting

**`FileNotFoundError: NCBI gene_info file not found`**
Run the `wget` + `gunzip` commands in the [Input Data](#input-data) section.

**`KeyError: 'spaceflight' not found`**
Notebook 1 must be run to completion (including Step 7, spaceflight module injection) before Notebook 2.

**`benchmark.json` is stale after rebooting or changing `N_ITER`**
Delete `data/output/benchmark.json` and re-run the benchmark cell in Notebook 2.

**Ray workers OOM-killed**
Reduce `N_WORKERS` manually by editing `benchmark.json` before re-running the screening cell. Alternatively, increase `RAM_SAFETY` margin or free system memory.

**Screening interrupted mid-run**
Simply re-run the screening cell (or `run_screening.py`). Completed checkpoints are detected automatically and skipped.
